Kolom Audit Penting yang Harus Disertakan dalam Tabel Basis Data Anda

Eugenius Wahyudiarto Jan 23, 2025
database

Saat mendesain tabel basis data, penting untuk menyertakan kolom audit guna melacak pembuatan, modifikasi, dan penghapusan rekaman. Kolom audit ini memberikan wawasan penting tentang perubahan data, meningkatkan akuntabilitas, dan membantu memenuhi standar regulasi atau kepatuhan. Dalam artikel ini, kami akan membahas kolom audit utama yang harus Anda pertimbangkan untuk ditambahkan ke tabel, termasuk stempel waktu, pelacakan pengguna, dan pembuatan versi, guna memastikan siklus hidup rekaman yang transparan dan dapat dipelihara.

Kolom audit / Audit fields biasanya mencakup kolom yang melacak perubahan atau kejadian dalam siklus hidup data. Kolom ini digunakan untuk memantau dan menjaga integritas data, sering kali untuk tujuan akuntabilitas, keterlacakan, atau kepatuhan. Berikut ini adalah daftar umum kolom audit:

Kolom Audit Standar

created_at

Melacak kapan rekaman dibuat.

Biasanya diatur ke stempel waktu saat ini pada saat penyisipan.

Tidak dapat diubah (tidak boleh diperbarui setelah pembuatan).

updated_at

Melacak waktu terakhir rekaman dimodifikasi.

Diperbarui secara otomatis setiap kali rekaman berubah.

Kolom Audit Opsional

deleted_at

Melacak kapan rekaman “dihapus sementara” (penghapusan logis tanpa menghapus data dari basis data).

Berguna dalam mekanisme penghapusan sementara di mana data disembunyikan alih-alih dihapus secara permanen.

Dapat dibatalkan secara default (null jika rekaman tidak dihapus).

created_by

Melacak ID atau nama pengguna pengguna/sistem yang membuat rekaman.

Berguna untuk mengidentifikasi asal rekaman.

updated_by

Melacak ID atau nama pengguna pengguna/sistem yang terakhir memperbarui rekaman.

Memastikan akuntabilitas atas modifikasi.

deleted_by

Melacak ID atau nama pengguna pengguna/sistem yang melakukan penghapusan lunak.

Kolom Audit Lanjutan atau Khusus-Kepatuhan

version

Melacak versi rekaman yang sering digunakan dalam.

optimistic_locking

Bertambah pada setiap pembaruan untuk mendeteksi konflik pada transaksi bersamaan.

change_reason

Mencatat alasan pembaruan atau penghapusan rekaman.

Sering digunakan dalam sistem yang memerlukan jejak audit yang ketat.

restored_at

Melacak saat rekaman yang dihapus sementara dipulihkan.

restored_by

Melacak ID pengguna/sistem yang memulihkan rekaman yang dihapus sementara.

Contoh Tabel dengan Kolom Audit

CREATE TABLE audit_example (
    id INT AUTO_INCREMENT PRIMARY KEY,
    data TEXT,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    deleted_at DATETIME DEFAULT NULL,
    created_by INT,
    updated_by INT,
    deleted_by INT,
    change_reason VARCHAR(255),
    version INT DEFAULT 1
);

Praktik Terbaik untuk Kolom Audit

Gunakan nilai dan pemicu default (jika diperlukan) untuk mengotomatiskan penandaan waktu dan pembaruan.

Simpan pengenal pengguna di created_by, updated_by, dll., untuk menautkan perubahan ke pengguna tertentu.

Normalisasikan bidang audit di semua tabel untuk menjaga konsistensi.

Gunakan penghapusan sementara (deleted_at) alih-alih penghapusan fisik untuk menghindari kehilangan data yang tidak disengaja.

Ini memastikan Anda dapat melacak tidak hanya “apa” dan “kapan” perubahan tetapi juga “siapa” dan “mengapa,” yang penting untuk sistem yang memerlukan jejak audit terperinci.
